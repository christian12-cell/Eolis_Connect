## 7. Page "Mot de passe oublié" — refonte complète

### Première version (intermédiaire, abandonnée)
Simple page avec champ email + téléphone, envoi d'un lien. Non fonctionnel car :
- Appel à `/api/auth/forgot-password` sans `apiUrl()` → Next.js API routes au lieu du backend
- Background blanc (manque du bg auth)
- Endpoint backend inexistant au départ

### Version finale — Flow multi-étapes sécurisé

**Flow :**
1. **Choisir le mode** : Email ou Téléphone (deux grandes cartes)
2. **Saisir la valeur** : Email ou téléphone
3. **Confirmation d'identité** : Le système cherche en base et affiche la valeur **masquée stockée** (ex: `an*****@gm***.c*`) — "Est-ce bien le vôtre ?"
   - OUI → envoi OTP
   - NON → ressaisir
4. **OTP** : Code à 6 chiffres (email ou SMS selon le mode), 10 min, max 3 tentatives, bouton renvoyer
5. **Confirmation username** : Affiche le username masqué (ex: `Ch******.DE***`) — "Est-ce bien votre compte ?"
   - OUI → envoi du lien de réinitialisation
   - NON → page support
6. **Succès** : Confirmation + avertissement ⚠️ "lien à usage unique, expire dans 48h"
7. **Compte introuvable** : "Essayer avec l'autre mode" + "Recommencer" + lien support
8. **Support** : Email support + lien recommencer

**Backend (`auth.py`) — 4 nouveaux endpoints :**
- `POST /api/auth/forgot-password/lookup` : cherche par email (case-insensitive) ou téléphone, retourne masked + lookupToken (JWT 15 min)
- `POST /api/auth/forgot-password/send-otp` : génère OTP, envoie via email (`send_otp_email`) ou SMS (`sms_otp`)
- `POST /api/auth/forgot-password/verify-otp` : vérifie le code, retourne maskedUsername + verifiedToken
- `POST /api/auth/forgot-password/send-reset` : génère le PasswordReset token 48h, envoie le lien

**Helpers backend :**
- `_mask_email(email)` : `christian@gmail.com` → `ch*****@g****.c*`
- `_mask_phone(phone)` : `+237690123456` → `+23****456`
- `_mask_username(username)` : `Christian.DENMEKO` → `Ch******.DE***`
- `_sign_lookup(user_id, mode)` / `_verify_lookup(token)` : JWT HMAC signé avec SECRET_KEY

**Stockage OTP :** Utilise le modèle `OtpCode` existant avec `phone = f"fp:{user_id}"` (39 chars, rentre dans String(50)) pour distinguer des OTPs classiques.

**email_service.py :** Nouvelle fonction `send_otp_email` — code affiché en grand (48px, monospace, letter-spacing 8px), bilingue.

**sms_service.py :** Nouvelle fonction `sms_password_reset`.

## 11. Double envoi de dossier au retour en ligne

**Problème :** Un dossier créé offline était envoyé **deux fois** au retour en ligne.

**Cause :** Deux systèmes de sync indépendants traitaient tous les deux `CREATE_TICKET` depuis la même queue IndexedDB :
- `MobileLayout` → `syncPending()` (offline-sync.ts) via l'event `online`
- `OfflineBanner` → sa propre fonction `flush()` avec la même logique

Les deux se déclenchaient simultanément au retour en ligne, lisaient le même `CREATE_TICKET` avant que l'un d'eux l'ait supprimé → double soumission.

**Fix 1 — Mutex** (`offline-sync.ts`) : Ajout d'un flag `_syncing` module-level. Si un sync est déjà en cours, tout appel supplémentaire retourne immédiatement `{ sent: 0, refs: [] }`.

**Fix 2 — Unification** (`OfflineBanner.tsx`) : Suppression de toute la logique `flush()` de l'OfflineBanner. Il appelle désormais directement `syncPending()`. Les types `AGENT_REPLY` et `INTERNAL_NOTE` ont été migrés dans `offline-sync.ts` pour centraliser toute la logique.

## 13. Délai 5-10s au chargement offline

**Problème :** Cliquer sur "Nouvelle demande" en mode offline prenait 5-10 secondes avant que la page soit utilisable — le browser attendait le timeout réseau des appels API.

**Cause :** `apiFetch` n'avait ni timeout ni vérification de `navigator.onLine`.

**Fix (`api-client.ts`) :** Deux améliorations :
- **Shortcut offline** : si `!navigator.onLine` et méthode GET → sert le cache IndexedDB immédiatement sans tenter le réseau (page instantanée en mode avion)
- **Timeout 8s** : `AbortController` avec 8 secondes max pour les connexions lentes — évite les attentes infinies

## 14. Notes internes — layout, scroll, notifications

### Layout droite/gauche
Vos propres notes → droite (`rounded-tr-sm`), icône 🔒 discrète. Notes des autres → gauche (`rounded-tl-sm`), nom de l'auteur visible.

### Scroll séparé par tab
Chaque tab (client / notes internes) a son propre état scroll + compteur non-lus indépendant :
- `prevClientLenRef` + `clientUnread` + `isClientScrolledUp`
- `prevInternalLenRef` + `internalUnread` + `isInternalScrolledUp`
- Badge rouge sur le tab "Notes internes" quand une nouvelle note arrive pendant qu'on est sur l'autre tab
- Changer de tab → scroll automatique en bas + reset du compteur
- **Initialisation** : les deux refs sont initialisées au chargement avec le compte exact (`msgs.filter(...)`) pour éviter de compter les messages existants comme "non lus"

### Auto-mark-read
Quand un nouveau message arrive via WebSocket (pendant que l'agent est sur la page), `mark-read` est appelé automatiquement → la cloche ne monte pas pour des choses déjà vues.

### Notification agent assigné sur note interne (`messages.py`)
L'agent responsable du dossier reçoit une notification "Note interne" même s'il n'est pas @mentionné, tant qu'il n'est pas déjà dans la liste des personnes notifiées.
